## Bekanntes Problem: expo-sqlite auf Web

`openDatabaseSync` wirft im Web-Build zuverlässig `Error: Sync operation
timeout` (bestätigt per Playwright, headless und headed, sowie manuell im
Browser). Ursache: die synchrone WASM/OPFS-Bridge (`wa-sqlite`) von
expo-sqlite im Web-Polyfill. Betrifft **nur die Web-Plattform** — nativ
(iOS/Android) gibt es diese Bridge nicht, dort läuft `openDatabaseSync`
direkt gegen die native SQLite-Engine. Da die App laut CLAUDE.md primär
offline-mobil ist, wurde das bewusst nicht weiter debuggt.

## Vorschau-Versuch (Troubleshooting-Protokoll)

Ziel: der Nutzerin die App auf einem echten Gerät zeigen. Verlauf:

1. **Web-Preview** (`expo start --web`): blockiert durch das oben genannte
   `Sync operation timeout`-Problem.
2. **LAN + Expo Go (QR-Code)**: Verbindung schlug fehl ("offline"). Ursache
   gefunden: Windows stuft das WLAN "Feiwel123 2" als **"Öffentlich"** ein →
   Firewall blockiert eingehende Verbindungen zum Metro-Server (Port 8081),
   obwohl der Server korrekt auf `0.0.0.0` lauschte. Nutzerin hat keine
   Admin-Rechte, Netzwerkprofil konnte nicht auf "Privat" umgestellt werden.
3. **Tunnel-Modus** (`expo start --tunnel`, `@expo/ngrok` als neue
   Dev-Dependency installiert): Tunnel kam zustande
   (`https://jemft0s-anonymous-8082.exp.direct`), umgeht das
   Firewall-Problem, da die Verbindung über das Internet statt direkt im LAN
   läuft.
4. **Bild-Übertragung**: Der generierte QR-Code (PNG, via `qrcode`-npm-Paket
   im Scratchpad-Ordner erzeugt) wurde im Chat nicht als Bild angezeigt
   (Client-Umgebung der Nutzerin rendert keine Bilder inline) → Datei stattdessen
   auf den Desktop kopiert (`C:\Users\minaa\OneDrive\Desktop\expo-qr-code.png`
   — Hinweis: Desktop ist bei diesem Profil OneDrive-umgeleitet, nicht
   `C:\Users\minaa\Desktop` direkt).
5. **SDK-Mismatch**: Expo Go (App-Store-Version) verlangte ein Update, obwohl
   die Nutzerin bereits die neueste installierte Version hatte. Ursache:
   Projekt nutzt Expo SDK 57 (sehr aktuell), die veröffentlichte Expo-Go-App
   hinkt Store-seitig oft hinterher und unterstützt jeweils nur eine
   SDK-Version gleichzeitig.

**Entscheidung:** Live-Vorschau auf dieser Session zurückgestellt. Optionen für
später: (a) Expo-Go-Beta über TestFlight, (b) SDK-Downgrade auf eine von der
aktuellen Expo-Go-Store-Version unterstützte Version, (c) echter EAS
Dev-Build (Teil von Phase 6 laut Plan — löst auch das expo-sqlite-Web-Problem
nicht, aber macht die Web-Preview ohnehin unnötig, da native SQLite genutzt
wird).
